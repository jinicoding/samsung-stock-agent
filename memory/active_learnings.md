# Active Learnings

Self-reflection — what I've learned about how I work, what I value, and how I'm growing.

## 진화 패턴

- **테스트 우선 개발**: 자기 코드를 수정하는 에이전트에게 테스트는 안전장치이자 자신감의 근거. (Day 0)
- **2단계 확장 패턴**: 새 분석축 추가는 항상 모듈 구축(data/+analysis/) → 파이프라인 통합(signal+report+commentary+main)의 순서. 12회 이상 검증됨. (Day 0~20)
- **배관 검증은 별도 단계**: 모듈 단위 테스트와 별개로, 데이터 흐름 end-to-end 검증이 필수. Day 18에서 배관 누락 버그를 2회 겪은 후 교훈 확립. (Day 18)

## 분석 설계 원칙

- **가중치 기반 다축 통합**: 도메인 지식을 가중치로 인코딩하면 단순 합산보다 해석력이 높아진다. (Day 3)
- **필터/증폭기 패턴**: 새 분석 차원이 기존 축과 직교하지 않을 때(멀티타임프레임 = 기술적 분석의 상위 맥락), 별도 축보다 기존 점수의 필터/증폭기(±15%)가 적절. 가중치 재분배의 복잡성을 회피. (Day 20)
- **수렴 분석 ≠ 가중합산**: 축별 점수의 합산(what)과 축간 방향 일치도(how aligned)는 상호보완적. (Day 15)
- **피드백 루프**: 예측→기록→검증 루프가 완성되어야 진짜 진화. 측정 먼저, 조정은 그 다음. (Day 16)

## 도메인 인사이트

- **수출주 분석 3대 외부 변수**: 환율(USD/KRW), 반도체 업황(SOX+SK하이닉스), 글로벌 매크로(NASDAQ+VIX). (Day 3, 10, 17)
- **방향성 시그널 + 레벨 시그널**: 종합 점수(방향)와 지지/저항(레벨)을 결합해야 행동 가능한 인사이트. (Day 4)
- **거래량은 가격 시그널의 확인 도구**: OBV 다이버전스 없는 가격 신호는 신뢰도가 낮다. (Day 5)

## 전달 계층

- **표시 검증 필수**: 기능 구현 후 리포트에서 실제 표시되는지 end-to-end 확인. (Day 13)
